# 機能的コード生成の原則

`typeship` は、`purescript-bridge`、`haskell-to-elm`、`servant-foreign`、および serde/aeson スタイルの一般的な派生と同じ広い形状に従います。境界を一度記述し、その記述をデータとして扱い、ターゲット言語のアーティファクトを決定論的にレンダリングします。

## ソースタイプが境界を所有する

Rust コマンドペイロードは真実の源です。TypeScript の宣言は、並行する IDL として維持するのではなく、Rust/serde ワイヤー形状から派生すべきです。生成されたファイルはコミットされますが、CI はそれを再レンダリングでき、ドリフトが発生した場合には失敗する必要があります。

## 製品と和が可視であること

Rust 構造体は製品タイプであり、TypeScript のオブジェクトレコードまたはインターフェースとしてレンダリングされます。Rust 列挙型は和タイプであり、ワイヤーフォーマットが許す限り、閉じた TypeScript のユニオンとして残るべきです。C のような列挙型の場合、文字列リテラルのユニオンはフロントエンドを網羅的に保ちます：

```ts
export type ConnectionStatus = "connected" | "idle";
```

ユニオンが閉じている場合、生成された `assertNever` ヘルパーは、消費者が新しい未処理のバリアントを TypeScript のコンパイルエラーに変えることを可能にします。

## 名前付けは契約である

Rust フィールドは `snake_case` のままです。JSON および TypeScript プロパティは一般的に `camelCase` です。そのマッピングを二つの無関係な文字列変換ではなく、単一の双方向ポリシーとして扱います。`WorkspaceSnapshot` のようなタイプは変更されずに通過します。

## エンコードとデコードは一致しなければならない

Serde 属性は、名目上の Rust タイプを変更することなく JSON 形状を変更できます。ブリッジはそれらの属性をモデル化するか、明確に警告する必要があります。重要な危険には以下が含まれます：

- `untagged` 列挙型、なぜならスイッチするための識別子がないからです；
- `flatten`、なぜならネストされたフィールドが親オブジェクトにスパイスされるからです；
- `skip_serializing_if` と `default`、なぜならキーが存在しない場合があるからです；
- `transparent`、なぜならラッパーがワイヤー上で消えるからです；
- 名前の衝突、なぜなら二つの Rust フィールドが同じ JSON キーにマッピングできるからです。

`diagnostics` モジュールは、これらの危険を一つの共有カタログに保持し、将来のバックエンドアダプターやドキュメントが同じ警告テキストを使用できるようにします。

## アセンブリはリフレクションから独立している

コアクレートは、`ts-rs`、`specta`、または `schemars` が Rust タイプをどのように発見するかを知る必要はありません。これらのツールは、Rust タイプを宣言文字列または小さな中間表現に変換できます。`typeship` は、その後、ヘッダー、インポート、コマンドラッパー、オプションのヘルパー、およびドリフトチェックを含む完全な TypeScript モジュールを組み立てます。
